# 终端模块 1.0 方案与 2.0（当前实现）对比及优化点

本文档约定：

- **1.0**：指 `doc/终端模块开发方案.md` 中描述的 **规划态**：M0/M1/M2 里程碑、分层目标、默认假设（含 `ghostty-surface`、CPU 光栅等）。
- **2.0**：指 **当前仓库实际演进**（以默认 `ghostty-vt` + egui/WGPU 集成为准），细节以 `doc/终端模块开发指南.md` 与源码为准。

二者不是「废弃 / 替代」关系：**1.0 仍是产品与技术愿景的纲**；**2.0 是在资源与路径约束下已落地的架构与优化结果**，部分条目超前于原文档，部分条目尚未达到原文档目标。

## 2. 渲染管线差异（核心）

### 2.1 1.0 规划

- 跨平台先 **CPU 画字再上纹理**，再交给 wgpu/egui 显示。
- macOS 可走 **IOSurface 抓屏** 一体化终端像素。
- 字体与设计规范对齐（JetBrains Mono / SF Mono 等）放在 M1 体验增强中。

### 2.2 2.0 实现

- **不在 CPU 上整屏 RGBA 再上传**作为主路径；改为：
  - **GPU**：清屏 → **背景 instance（每 cell）** → **字形 instance（采样 R8 atlas）**，结果写入 **RGBA8 离屏**，再 **egui `TextureId` + `Shape::image`**。
  - **CPU 回退**：galley + 按 **脏行** 重建 cache；字体跟随 egui 默认 monospace（与 GPU atlas 字体可不一致）。
- **样式**：反色在 `**ghostty_vt` 快照阶段** 已交换 fg/bg；GPU 片元级实现 **粗体 / 下划线**（空格仅下划线仍发 quad）。
- **稳健性**：字体加载或 **拉丁轮廓探测** 失败时 **自动关闭 GPU 正文**，避免「有连接无字形」。

## 3. I/O 与交互


| 项目                 | 1.0                      | 2.0                                                       |
| ------------------ | ------------------------ | --------------------------------------------------------- |
| 阻塞                 | 明确禁止 UI 线程无限 read        | `session_bridge` **预算** + 分块 `on_bytes`                   |
| 异步模型               | 推荐独立 read task + channel | 当前以 **会话非阻塞 read + 每帧聚合** 为主（实现更简单，与现有 `AsyncSession` 契合） |
| TerminalController | 集中键盘/鼠标/剪贴板策略            | 逻辑仍多在 `**terminal_view`** 内，**与 1.0 的 Controller 拆分有差距**  |

## 4. 2.0 相对 1.0 的「已落实优化点」

以下可视为在 1.0 文档目标之外的 **工程化增值**（或提前完成原文档 M1/M2 中的部分内容）：

1. **绘制提交减负**：主路径由海量 `egui::Painter::text`/布局改为 **GPU instance**，火焰图瓶颈从 CPU 字形布局侧转移（启用 GPU 且字体可用时）。
2. **增量 FFI / 数据**：**脏行** 更新 + styled run 复用缓冲，降低每帧全表扫描与分配（见 `ghostty_vt` 与 `vt_widget` 快照逻辑）。
3. **I/O 预算**：爆发输出时 **单帧字节/读次数上限**，对齐 1.0「不卡 UI」原则，且无需先引入完整 tokio 管线即可见效。
4. **中英混排与 TTC**：**自建 glyph atlas**（Songti + 多路径回退 + `RUST_SSH_FONT_`*），较 1.0 文档中的「8×8 CPU」更接近真实终端观感。
5. **可观测性**：`term-prof`、周期性计数器，支撑 **数据驱动** 决策（先减 draw 还是先减 FFI）。
6. **健壮回退**：`RUST_SSH_TERMINAL_GPU=0`、图集不可用时 **自动 CPU 正文**，避免静默空白。

## 5. 尚未完全达到 1.0 文档的部分（技术债 / 后续方向）

若要对齐 1.0 路线图，仍可排期：

1. `**ghostty-surface` / IOSurface**：当前仓库 **未接入**；若需 macOS 原生一体化渲染，仍属 M2 量级工作。
2. `**TerminalController` 独立模块**：减少 `terminal_view.rs` 体积，统一输入与策略（剪贴板、快捷键、后续鼠标扩展）。
3. `**TerminalEngine` / `TerminalRenderer` 与后端绑定**：让 `GhosttyVtBackend` 显式实现组合 trait，便于第三套引擎/渲染接入与单元测试替身。
4. **tokio mpsc read loop**：在极高吞吐场景下可进一步 **解耦 UI 与网络线程**（1.0 推荐形态）。
5. **设计规范逐项验收**：滚动条样式、选定 Accents、指定 UI 字体族与行高等，部分仍分散在 egui 主题与常量中，需与 `组件设计规范` 对表。
6. **GPU 侧能力扩展**：斜体、删除线、闪烁、atlas 耗尽策略（LRU/多页）、宽字符与字素簇 key 一致化等。
