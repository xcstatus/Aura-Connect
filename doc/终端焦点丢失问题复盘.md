# 终端丢失焦点问题复盘

## 背景

在终端区域获得键盘焦点后，按 **Tab** 或 **方向键（↑↓←→）** 时，终端会出现“焦点被移走”的现象，表现为：

- 终端不再接收键盘输入（或需要再次点击终端才能继续输入）。
- 在某些场景下，焦点被 UI 中的其它可交互控件夺走（例如文本输入框、按钮等）。

该问题会严重影响终端交互体验，尤其是 shell 常用的 **↑/↓**（历史命令）和 **Tab**（补全）。

## 现象与复现条件

- **必现操作**：终端获得焦点后，按 `Tab` / `↑` / `↓` / `←` / `→`。
- **复现概率**：与当前 UI 中“是否存在其他可聚焦控件”高度相关；控件越多，越容易观察到焦点被切走。

## 根因分析

### 1) egui 的默认焦点导航行为

在 egui 中，**Tab/方向键**不仅仅是普通按键事件，它们还参与 **焦点导航（focus navigation）**：

- egui 会在帧开始阶段根据这些按键决定“将焦点移动到下一个/附近的可聚焦控件”。
- 这意味着：即使终端想把这些按键当作“终端输入”，egui 也可能先把它们用于“切换焦点”，导致终端失焦。

### 2) 终端实现方式导致被纳入焦点系统，但未锁定导航键

终端组件通过 `ui.interact(rect, id, Sense::click())` 创建了可交互区域，并在点击时请求焦点。
但在修复前，终端没有对 Tab/方向键做“焦点锁定过滤”，因此会触发 egui 的默认导航。

### 3) 曾经的“抢焦点/吞键”方案不推荐

在排查过程中，曾尝试过两类“强行修复”：

- **抢回焦点**：检测到失焦后立刻 `request_focus` 抢回来。
- **吞键**：在终端有焦点时主动 `consume_key(Tab/Arrow...)`。

这些方式存在明显副作用（影响全局输入状态、与其他输入控件冲突、可维护性差），因此最终被移除。

## 排查过程摘要

### 1) 搜索“谁在 request_focus”

全仓检索 `request_focus`，确认是否存在“其他组件无意中抢焦点”的情况。

结论：存在少量合理场景（例如 quick connect 输入框首次弹出自动聚焦），但核心问题仍是 **egui 默认焦点导航**。

### 2) 校验 egui 推荐的解决路径

在 `egui 0.33.3` 中，官方提供了针对焦点导航的机制：

- `Memory::set_focus_lock_filter(id, EventFilter)`

它用于控制一个已聚焦的 widget 是否会因为 Tab/方向键/Esc 而“丢失焦点/参与焦点切换”。

## 最终修复方案

### 方案原则

- **终端点击获得焦点**：保留。
- **终端持有焦点时，锁定 Tab/方向键等导航键**：使用 egui 官方机制 `set_focus_lock_filter`。
- **移除抢焦点/吞键等 hack**：避免副作用。

### 代码变更点

- 文件：`src/ui/components/terminal_view.rs`
- 行为：
  - 点击终端：`mem.request_focus(terminal_focus_id)`
  - 终端有焦点：`mem.set_focus_lock_filter(terminal_focus_id, EventFilter { tab: true, horizontal_arrows: true, vertical_arrows: true, escape: true })`
  - 删除：
    - `had_focus_last_frame` 相关“抢回焦点”逻辑
    - `consume_key` 相关“吞键”逻辑

## 为什么该方案有效

`EventFilter` 的语义是：当 widget 已聚焦时，指定哪些按键事件将被该 widget **独占处理**，不再触发 egui 的焦点导航。

因此：

- `Tab` 不再触发“切换到下一个可聚焦控件”
- `方向键` 不再触发“焦点移动到邻近控件”
- 终端可以稳定地把这些按键作为“终端输入”处理

## 回归验证

- `cargo check` 通过
- 手工验证：
  - 终端聚焦后连续按 `↑/↓` 不失焦
  - 终端聚焦后按 `Tab` 不失焦
  - 在界面存在其他可聚焦控件时（如 quick connect 输入框、表单输入框），终端仍能保持焦点，除非用户明确点击其他控件

## 后续建议

- **对“终端区域”这种输入密集组件**：优先使用 `set_focus_lock_filter`，而不是靠 `request_focus` 反复抢回。
- **减少全局吞键**：`consume_key` 适合局部、明确的输入控制，不应作为解决焦点导航的常规手段。
- **对自动聚焦的输入框**：保持“一次性聚焦”（focus-once）模式，避免每帧反复 request_focus 抢占终端焦点。
