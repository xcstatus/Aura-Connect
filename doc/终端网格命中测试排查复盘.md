# 终端网格命中测试排查复盘

记录 Iced 样式终端视口中，**鼠标点击列/行**与 **PTY 网格**对齐问题的现象、根因与排查顺序。便于后人遇到类似「选中/拖选错位」时对照。

**关联文档**

- 架构边界与 SSOT（含与本题相关的 **§5.1 实践约束**）：[`iced能力边界与终端重构指南.md`](iced能力边界与终端重构指南.md)
- 终端模块进度与路径说明：[`进度/终端模块.md`](进度/终端模块.md)

---

## 1. 现象

### 1.1 纵向

- 在同一列上从上到下点击多个字符（例如一列 `d`），**行号跳跃**或视觉行距约为「两行」一根网格线。
- `term_hit_test` 日志里 `ch` 与 `local_y` 换算合理，但**肉眼行距**不等于 `ch`。

### 1.2 横向

- 同一提示行上从左到右点字符（例如 `dream…` 或 `dr` 与 `28k`），**左侧**列号尚接近正确，**越往右** `cell`/`ideal_f` 与眼睛看到的字符差得越多（**累积偏移**）。
- 仅把命中测试从 `metric_cw` 改为 `w_map/cols` **不足以**完全消除字粒度点击偏差。

---

## 2. 目标与约定

- **单一真相源**：视口几何在 `terminal_viewport.rs`（`TerminalViewportSpec`、`terminal_scroll_area_rect`、`window_point_to_grid_with_dims`）。
- **列**：在可映射宽度 `w_map`（扣除右侧滚动条命中豁免带）上 **均分为 `cols` 段**，`cell_w = w_map / cols`。
- **行**：`ch = spec.term_cell_h()`，与 PTY 行高及 `terminal_rich` 每行固定高度一致。

---

## 3. 排查流程（时间顺序）

### 步骤 A — 确认命中公式与视口矩形

1. 用 `RUST_LOG=term_hit_test=debug` 打出 `origin`、`local`、`cell_w`、`metric_cw`、`rect_wh`。
2. 对照 `terminal_scroll_area_rect`：窗口坐标是否从 **面板 padding + 内边框** 起算，与真实绘制区域左/上边缘一致。
3. 确认列向使用 **`w_map/cols`** 与 PTY `cols` 同源（`state.active_terminal().grid_size()`）。

**结论**：横向公式与视口整体合理后，仍存在「越往右越偏」，说明问题更多在 **渲染排版**，而非单纯 `local_x` 换算。

### 步骤 B — 纵向：行高与 `rich_text`

1. 怀疑 `rich_text`/`Paragraph` 的 **`min_bounds` 行高**大于名义 `ch`，导致**视觉行间距**约 `2×ch`，而 hit-test 仍按 `ch` 分桶。
2. 在 `terminal_rich.rs` 用 **`fixed_terminal_row`**：`container(inner).height(Fixed(line_px)).width(Fill).align_y(Top)`，其中 `line_px = term_cell_h()`。

**结论**：纵向错位主要来自 **Iced 行盒高度** 与 **PTY `ch`** 不一致；锁行高后与 `floor(local_y/ch)` 对齐。

### 步骤 C — 横向：合并 span 与 shaping

1. 怀疑多格合并成一段字符串（如 `"dream"`）整体 shaping，**字内 advance 非「一格一等宽网格」**，导致 `floor(local_x/cell_w)` 与逻辑列不一致。
2. 将 `append_cells`、`build_row_fragments` 改为 **每单元格一段**（每格一 fragment / 一 span），减轻聚类 shaping。

**结论**：有改善，但若仍是一条 **单行 `rich_text` 多 span**，水平方向仍按**真实字宽累加**，与 **强制等宽列带** 长期仍会 **累积偏差**。

### 步骤 D — 横向定版：每格固定像素宽

1. 提取 **`terminal_scroll_cell_geometry`**：`w_map` 与 **`cell_w`** 与 `window_point_to_grid_with_dims` **完全一致**。
2. 在 `terminal_main_area` 用当前 `window_size` + `grid_size().0` 计算 **`cell_w_hit`**，传入 `styled_terminal`。
3. **`rich_single_span_cell`**：每格 `rich_text([单 span])` 外包 **`container(width: Fixed(cell_w_hit), height: Fixed(line_h))`**，`align_x(Left)`。
4. **`styled_row_line` / `styled_row_line_from_fragments`**：用 **`Row::with_children(...).spacing(0)`** 拼行，去掉「整行一条 rich_text」布局。

**结论**：**渲染列宽**与**命中列带**同源，`local_x / cell_w` 与屏幕上第 `k` 格左边界一致，**累积漂移消除**。

---

## 4. 根因对照表

| 问题 | 根因 | 修复要点 |
|------|------|----------|
| 纵向错位 | `rich_text` 行盒高度 ≠ `ch` | `fixed_terminal_row` 固定 `term_cell_h()` |
| 横向跳列（整段字母） | 多格合并成一段 shaping | 每格独立 fragment/span |
| 横向越右越偏 | 多 span 水平累加字宽 ≠ `cols` 等分带宽 | 每格 `Fixed(cell_w)` + `Row`；`cell_w` 来自 `terminal_scroll_cell_geometry` |

## 6. 后续注意点

- **性能**：每行 `cols` 个子控件（`Row` + `container` + `rich_text`），大窗口需观察帧时间与滚动流畅度；若吃紧可考虑视口内行缓存、脏行重建策略（现有 fragment 快路径仍保留）。
- **极宽字符**：单格内 glyph 宽于 `cell_w` 时可能 **视觉溢出** 或被裁切，与真实终端「占两格」的 continuation 格需继续保持一致（continuation 仍占独立一格容器）。
- **Plain 模式**：本复盘主要针对 **Styled + `terminal_rich`**；纯文本路径若也要精确选中，需单独核对是否同样要 **固定字格宽**。

---

## 7. 验证建议

1. 开 `RUST_LOG=term_hit_test=debug`。
2. 同一行从左到右点 5～10 个已知字符，确认 `cell` 列号连续、与 shell 提示符上字符位置一致。
3. 同一列从上到下多点几行，确认行号与视觉行一致、无「双倍行距」感。
