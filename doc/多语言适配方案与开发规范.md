# RustSsh 多语言适配方案与开发规范

## 1. 目标与范围

本文档用于指导 RustSsh 的多语言（i18n）落地，目标是：

- 支持 `zh-CN` / `en-US` 的完整 UI 文案切换。
- 避免业务代码中继续新增硬编码文案。
- 建立可持续维护的多语言开发规范（命名、结构、提交流程、测试标准）。

当前优先范围：

- `src/ui/**` 全量界面文案
- UI 弹窗提示、Toast、按钮、表头、空态提示
- 与 UI 强相关的状态文案（连接状态、重启提示等）

暂不优先：

- 日志（`log::info!/warn!`）多语言化（日志可先保留英文或中文，后续统一）
- 协议层/底层错误码的本地化（先通过 UI 层映射）

---

## 2. 现状分析（基于当前代码）

### 2.1 已有基础

1) 已存在语言配置字段  
`src/settings.rs` 中 `GeneralSettings.language` 默认值为 `zh-CN`，且设置页可切换 `zh-CN` / `en-US`。

2) 字体已考虑中文显示  
`src/app.rs` 中初始化了中文字体 `Songti.ttc`。

### 2.2 关键问题

1) UI 文案大量硬编码  
`src/ui/components/modals.rs`、`src/ui/session_editor/*`、`src/ui/settings/*`、`src/ui/components/password_modal.rs` 等文件内存在大量中文字符串直写。

2) 数据层混入展示语言  
`src/app.rs` 中 `proxy_type` 默认值为 `"无"`，以及部分默认分组文案 `"未分类"` 直接进入业务流程，导致语言切换后数据与 UI 语义耦合。

3) 未建立统一翻译入口  
当前没有统一 `t("key")` 或等价接口，新增功能容易继续硬编码。

4) 语言切换不具备一致策略  
设置页可修改语言值，但 UI 文案未由语言值驱动重绘输出。

### 2.3 结论

项目已经具备“语言配置入口”，但缺少“翻译资源 + 统一访问接口 + 规范约束”，属于“可快速接入 i18n”的阶段。

---

## 3. 设计原则

1) **业务与文案解耦**  
业务状态使用枚举/结构，不使用中文或英文作为状态值。

2) **统一入口**  
所有用户可见文案必须通过 `i18n::tr(key)`（或同等接口）获取。

3) **键稳定、文案可变**  
代码依赖翻译键，文案可迭代，不影响逻辑。

4) **先 UI 后错误细分**  
第一阶段先覆盖 UI 静态文案与常见提示，再逐步细化错误码翻译。

5) **默认可回退**  
缺失翻译时回退到 `zh-CN` 或翻译键本身，并记录日志，避免空白 UI。

---

## 4. 推荐落地架构

建议新增：

- `src/i18n/mod.rs`
- `src/i18n/locale.rs`
- `src/i18n/catalog_zh_cn.rs`
- `src/i18n/catalog_en_us.rs`

示意：

- `Locale`: `ZhCn | EnUs`
- `I18n`: 持有当前 locale，提供：
  - `tr(&self, key: &'static str) -> &'static str`
  - `tr_fmt(&self, key: &'static str, args: &[(&str, &str)]) -> String`

在 `RustSshApp` 中增加：

- `pub i18n: crate::i18n::I18n`

初始化时从 `settings.general.language` 创建 locale；设置页变更语言后同步更新 `app.i18n` 并 `ctx.request_repaint()`。

---

## 5. 翻译键命名规范（必须遵守）

### 5.1 命名格式

统一使用小写点分层级：

- `settings.general.language.label`
- `settings.security.biometrics.title`
- `session_editor.general.host.required`
- `modal.connection.auth.title`

### 5.2 层级建议

- `common.*`：通用按钮/状态（保存、取消、确认）
- `header.*`
- `settings.general.*` / `settings.terminal.*` / `settings.security.*`
- `session_editor.general.*` / `session_editor.advanced.*`
- `modal.connection.*` / `modal.password.*`
- `toast.*`

### 5.3 禁止事项

- 禁止把整句中文当 key。
- 禁止使用动态 key（例如 `format!("toast.{}", x)`）。
- 禁止在 UI 代码中新增硬编码可见文案。

---

## 6. 文案参数化规范

动态文案用占位符，不拼接自然语言：

- key: `status.connecting_to`
- zh-CN: `连接中: {user}@{host}:{port}`
- en-US: `Connecting: {user}@{host}:{port}`

要求：

- 占位符仅用于变量（`{user}`、`{host}`）
- 不在业务层做中英文拼接
- 由 `tr_fmt` 统一替换

---

## 7. 业务模型改造规范

### 7.1 禁止业务字段直接存储本地化文本

当前示例问题：

- `proxy_type: String` 使用 `"无" | "SOCKS5" | "HTTP"`

建议改造为：

- `enum ProxyType { None, Socks5, Http }`

展示时再通过 i18n 映射：

- `proxy.none`
- `proxy.socks5`
- `proxy.http`

### 7.2 持久化字段保持语言无关

JSON/配置/数据库中存储稳定值（枚举值、代码值），不存储界面文案，防止切换语言后脏数据。

---

## 8. 分阶段实施计划

### 阶段 1（基础设施）

1) 新增 `i18n` 模块与双语词典。
2) 在 `RustSshApp` 挂载 `i18n` 实例。
3) 设置页语言切换时更新 `app.i18n` 并触发重绘。
4) 引入最小回退策略（key 缺失日志 + fallback）。

### 阶段 2（高频 UI 覆盖）

优先改造：

- `src/ui/settings/*`
- `src/ui/components/modals.rs`
- `src/ui/session_editor/*`
- `src/ui/components/header.rs`
- `src/ui/components/password_modal.rs`

原则：

- 每改一个模块，同步补齐 key 与双语条目。

### 阶段 3（状态与提示统一）

1) 连接状态、Toast、校验错误统一走 `i18n`。
2) 将硬编码业务文案改为枚举/常量值（如分组默认值、代理类型）。

### 阶段 4（质量门禁）

1) 新增 CI 检查（可选脚本）：
   - 扫描 `src/ui` 中是否存在新增中文硬编码。
2) 评审规范生效：PR 必须补齐 key 与双语词典。

---

## 9. 开发流程规范（PR 级）

每个涉及 UI 的 PR 必须满足：

1) 新文案先定义 key，再写词典。
2) `zh-CN` 与 `en-US` 同时提交，不允许只改一种语言。
3) 不在视图代码中新增直写文案（调试临时文案除外，提交前清理）。
4) 若新增业务状态，先定义枚举，再映射翻译键。
5) 自测至少覆盖两种语言切换下的主要交互路径。

---

## 10. 测试与验收标准

### 10.1 功能验收

- 切换 `zh-CN` / `en-US` 后，设置页、会话编辑、连接弹窗、密码弹窗文案正确切换。
- 动态文案占位符替换正确（host/user/port 等）。
- 缺失翻译时不出现空白文本。

### 10.2 回归验收

- 保存设置后重启应用，语言设置可持久化。
- 会话保存、连接、密码修改流程行为不受影响。
- 关键按钮含义在两种语言下均清晰一致。

---

## 11. 参考实现建议（简化版）

可先使用 `match` 静态映射，不强依赖外部 crate：

- 规模较小时维护成本低
- 编译期可发现部分拼写错误（配合 key 常量）

后续若词典规模明显增长，再评估引入：

- `phf`（静态映射）
- `fluent`（更强参数化与语法能力）

原则：先可用，再演进。

---

## 12. 团队约束（最终版）

从本文档生效日起，新增/修改 UI 功能遵循以下硬性规则：

1) 用户可见文案一律走 i18n。  
2) 禁止在业务模型中存储本地化文本。  
3) 双语词典必须同步更新。  
4) PR 未满足以上规则，不予合并。  

---

## 13. 建议的首批改造清单（按优先级）

1) `src/ui/settings/mod.rs` 与 `src/ui/settings/general.rs`  
2) `src/ui/components/modals.rs`  
3) `src/ui/session_editor/mod.rs` + `src/ui/session_editor/general.rs`  
4) `src/ui/components/password_modal.rs`  
5) `src/app.rs` 状态栏与重启提示文案  

完成以上后，项目即可进入“持续双语开发”状态。
