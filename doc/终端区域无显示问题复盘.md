# 终端区域无显示（黑屏/空白）问题复盘

本文档用于沉淀 RustSSH 在开发终端模块过程中出现的“**连接成功但终端区域无任何显示**”问题的原因、排查过程与最终处理方案，避免同类问题再次发生。

> 适用范围：`ghostty-vt` 后端 + `egui/eframe(wgpu)` 渲染路径（本仓库当前主线）。

---

## 1. 现象描述（Symptoms）

### 1.1 用户侧表现

- SSH 连接成功（认证成功、UI 提示已接收 session）
- 终端区域黑屏/空白，没有提示符、没有输出
- 有时伴随 CPU 占用高（持续刷新或重绘）

### 1.2 日志侧表现（关键证据）

- 后端能收到服务端数据（例如 `ChannelMsg Data: N bytes`）
- `AsyncSession::read_stream` 会打印“推送给 UI 渲染层”的日志
- 说明 **网络层与协议层正常、数据确实到达了应用侧**

---

## 2. 影响范围（Impact）

- 终端核心功能不可用（无法看到输出，无法完成交互）
- 由于终端是 RustSSH 的主工作区，影响几乎等同于“应用不可用”

---

## 3. 根因分析（Root Cause）

本次“终端区域无显示”在排查过程中经历了多次假设与收敛。最终确认：**渲染链路本身是通的，真正卡住的是 ghostty-vt 的 render state snapshot API 调用方式错误**，导致我们永远拿不到字符网格（grid）去光栅化，因此只能显示“空贴图/空帧”。

### 3.1 最终根因：`ghostty_render_state_get(... ROW_ITERATOR ...)` 的 out 参数传法错误（FFI）

`ghostty-vt` 的 render state 读取链路（官方示例）是：

`render_state_update -> row_iterator_new -> render_state_get(ROW_ITERATOR, &row_iter) -> row_iterator_next -> row_get(CELLS, &cells) -> row_cells_next -> row_cells_get(GRAPHEMES_*)`

我们在 Rust 侧一开始把 `ghostty_render_state_get(..., out)` 的 out 参数错误地写成了：

- 传入 `self.row_iter as *mut c_void`（把“句柄值”当成了 out 指针）

而官方 C 示例要求的是：

- 传入 `&row_iter`（指向“句柄变量”的指针，供函数写回/填充）

因此 `ghostty_render_state_get(row_iterator)` 返回 `-2 (GHOSTTY_INVALID_VALUE)`，表现为：

- `snapshot_codepoints_grid_and_clear_dirty()` 永远 `Err`
- `VtTerminalWidget` 永远拿不到 grid
- `frame.w/h/rgba` 一直是 `0/0/empty`（UI 上就是黑屏/空白）

最终修复：把 out 参数改为 `&mut self.row_iter`（以及 `row_get(CELLS)` 同理改为 `&mut self.row_cells`）。

### 3.4 补充（2.0 演进）：`update_render_state` 后全局 `dirty == FALSE` 导致「有光标无文字」

现象与本文档 **FFI 全失败** 不同：快照 API 正常，**光标**随 `cursor_state()` 更新，但 **ls 等输出不显示**；`RUST_SSH_TERMINAL_GPU=0` 走 CPU galley 时也同样无字（说明不仅是 GPU）。

根因：`paint_terminal` 在拉取 styled 行前会先调用 `ghostty_render_state_update`。部分情况下同步后 **全局** `ghostty_render_state_get(DIRTY)` 已为 `DIRTY_FALSE`，而 **行级 dirty 位** 未覆盖全部变更。下一逻辑若仅按「全局 dirty + `only_dirty=true`」增量遍历，会 **跳过所有行**，`cached_rows` 不更新；光标仍读最新 render_state → **只看见光标移动**。

处理：凡本帧已执行 `update_render_state`（例如 SSH/PTY 写入后请求的 `needs_render_update`），若此时全局 dirty 为假，则 **对该帧使用 `only_dirty=false` 全量重扫行**，再恢复纯 dirty 驱动以控制成本。实现见 `VtTerminalWidget::paint_terminal`（`refreshed_render_state_this_frame` / `only_dirty_rows`）。

### 3.2 重要证据：棋盘格验证“绘制链路 OK，内容源为空”

为二分“没画出来” vs “没产出帧”，我们在 `VtTerminalWidget` 增加了一个 debug 路径：当 RGBA 帧为空时绘制紫色棋盘格测试纹理。

现象：

- **棋盘格可见**：证明 `egui -> texture upload -> painter.image` 的渲染路径正常
- 同时 HUD 显示 `VT-FRAME 0x0 rgba=0`：证明问题在“拿不到 VT 网格内容”，而不是渲染被遮挡/被裁剪

并且 HUD 进一步显示 `VT-SNAPSHOT-ERR ghostty_render_state_get(row_iterator) failed: -2`，直接锁定根因。

### 3.3 次要因素（曾经的假设/干扰项）

这些因素会让现象更像“黑屏”，但不是最终根因：

- **首帧 cols/rows 为 0 导致 VT 未创建**：会让 ingest_output 被吞掉（已通过 clamp + 早创建规避）
- **强制 60fps repaint**：会拉高 CPU 并掩盖事件驱动 repaint（已移除）
- **自定义 wgpu pipeline/target format 不匹配**：曾经怀疑 callback 有调用但不出图；后来通过“棋盘格/TextureHandle”路径证明主要问题不在这里

### 3.2 诱因/放大器（Contributors）

虽然最终根因在 VT 贴图更新策略，但排查中还遇到了多个会造成“看起来像黑屏”或“加重问题”的因素：

- **首帧 cols/rows 为 0 导致 VT 未创建**：如果 VT 未创建，后续 `ingest_output` 会被吞掉。
- **强制 60fps repaint**：持续 `request_repaint_after(16ms)` 会拉高 CPU，并掩盖“事件驱动 repaint”逻辑缺失的问题。
- **旧 surface/IOSurface 路线覆盖**（曾存在）：不同渲染分支可能导致 vt 结果被遮挡（后来已清理为 VT-only）。
- **plain 模式误用 TextEdit 可编辑**：会出现“键盘输入能显示但不是服务端回显”的假象（后来改为只读，并最终移除临时方案）。

## 4. 排查过程（Debugging Timeline / Method）

排查采用“从外到内、逐层二分”的方式，核心思路是：**先证明数据有没有到、再证明解析有没有做、最后证明渲染有没有出图**。

### 4.1 第一步：验证连接与数据是否到达应用

证据：

- `ChannelMsg` 日志持续出现
- `AsyncSession` 打印 push-to-UI

结论：**SSH 协议层 OK**。

### 4.2 第二步：修复 UI 卡顿/高 CPU 干扰

措施：

- 去掉强制 60fps repaint 轮询
- 限制每帧 drain 的读取预算（避免 UI 阻塞）

目的：减少噪声，确保排查“黑屏”不是被卡顿掩盖。

### 4.3 第三步：加入诊断 HUD / 测试纹理进行链路二分（关键手段）

我们引入过以下诊断手段（后续已移除）：

- **VT selftest 注入行**：不依赖远端回显，用本地写入验证 VT 渲染链路是否通
- **Plain egui overlay 文本**：不走 callback，验证 UI 基础绘制是否正常
- **Plain 镜像输出**：把接收的 bytes（或 VT plain 行）直接用 egui 文本画出来，验证“数据确实存在”

通过这些诊断，我们得到关键结论：

- 能看到 `TERM-RECT` 红框：UI 确实在画终端区域（不是布局/裁剪/遮挡）
- 能看到紫色棋盘格：纹理上传 + `painter.image` 路径正常（不是 wgpu pass 没执行）
- 看到 `VT-SNAPSHOT-ERR ... -2`：问题明确在 ghostty-vt 的 render_state snapshot 调用

### 4.4 第四步：对照 ghostty 官方示例确认 API 调用方式（定案）

对照 `resource/ghostty/example/c-vt-render/src/main.c`，我们确认：

- `ghostty_render_state_get(... ROW_ITERATOR ...)` 必须传 `&row_iter`
- `ghostty_render_state_row_get(... CELLS ...)` 必须传 `&cells`

而 Rust 侧原实现传参方式不一致，导致 `GHOSTTY_INVALID_VALUE (-2)`。

---

## 5. 处理方案（Fix / Resolution）

### 5.1 中间阶段的临时方案（用于二分，已弱化/可移除）

为尽快恢复可用性，曾短期启用 “plain VT 文本模式” 作为主显示（不走 callback）。

该方案的价值是：

- 立即验证网络与解析链路正确
- 让终端先可用

但缺点是：

- 不能正确处理光标移动、清屏、局部刷新等终端语义（纯文本不可替代真正终端渲染）

### 5.2 最终修复方案（主线）：修正 ghostty-vt render snapshot 的 FFI out 参数

核心修复点位：`src/backend/ghostty_vt.rs`

修复内容：

- `ghostty_render_state_get(... ROW_ITERATOR ...)` 的 out 改为传 `&mut self.row_iter`
- `ghostty_render_state_row_get(... CELLS ...)` 的 out 改为传 `&mut self.row_cells`

修复效果：

- `snapshot_codepoints_grid_and_clear_dirty()` 能返回 grid
- `VtTerminalWidget` 能正常光栅化并生成 RGBA 帧
- 终端区域从“空白/黑屏”恢复为正常输出

### 5.3 配套增强（诊断与稳定性）

这些改动不一定全部需要长期保留，但它们在本次排查中极其关键：

- `TerminalView` 红框 `TERM-RECT`：确认终端区域是否可见、是否被裁剪
- `VtTerminalWidget` 紫色棋盘格：确认渲染链路是否正常（内容源为空时也能看到）
- `VT-SNAPSHOT-ERR` HUD：把 FFI 错误直接暴露在 UI 上，避免只看日志

另外，之前为降低噪声做的稳定性改动依然建议保留：

- `src/main.rs` 强制 `eframe::Renderer::Wgpu`（确保 wgpu callback 路径可用）
- `TerminalSessionBridge` 做 per-frame drain 预算，避免 UI 阻塞造成的假黑屏
- 清理旧版 ghostty-surface/IOSurface 路线，避免渲染分支遮挡（VT-only）

---

## 6. 如何验证修复（Verification）

### 6.1 手工验证

- 连接 SSH 后应立即看到：
  - 提示符（例如 `user@host ~`）
  - 输入命令回显
  - 服务端输出（`ls`, `pwd`, `echo`）

### 6.2 自动化/单元测试建议

当前已有与本问题相关的测试/保障点：

- `TerminalSessionBridge` 的 drain 预算与 resize 去抖单元测试：避免 UI 卡死/读写异常
- `VtTerminalWidget` 的 version bump 测试：确保内容变化会触发版本变化

已补充（建议/现状）：

- 为 `GhosttyVtTerminal` 增加单元测试，覆盖：
  - new/write_vt 后能 snapshot plain lines
  - new/write_vt 后能 snapshot codepoints grid（len 正确且包含非 0 codepoint）
  - resize 后 snapshot 仍可用
  - encode_key 对 Enter 能产出字节序列

---

## 7. 后续行动（Follow-ups）

为了避免回归与提升终端渲染质量，建议：

- **把渲染模式做成可配置开关（仅 Debug）**：GPU/CPU/Plain 等，方便快速二分
- **把关键诊断指标加入 Debug HUD（可开关）**：
  - drained bytes、rasterized_version、uploaded_version、纹理大小
- **逐步替换 CPU 字体光栅化**：当前是 font8x8 8x8 bitmap + 放大；后续可换成字体 atlas + GPU 文本渲染以获得真实终端观感与性能
