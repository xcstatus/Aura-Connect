# 终端模块开发方案（基于 ghostty）

本文档用于指导 RustSSH 的“终端模块（Terminal Module）”后续开发与落地实现（可视为 **1.0 级规划**）。与 **当前代码实装（2.0 演进）** 的差异与优化点见 **`doc/终端模块_1.0与2.0方案对比与优化点.md`**。内容结合：

- `doc/组件设计规范.md` 的终端视觉/交互规范
- `doc/系统架构.md` 的分层架构与性能目标
- 当前代码实现：`src/backend/ghostty_ffi.rs`、`src/backend/ghostty_vt.rs`、`src/terminal/*`、`src/ui/components/terminal_view.rs`

---

## 1. 目标与约束

### 1.1 目标（要交付的能力）

- **高性能渲染**：在 UI（egui/eframe）内嵌终端，目标 ≥ 60fps（参见 `doc/系统架构.md` 3.5）
- **跨平台可运行**：至少提供一个跨平台后端（`ghostty-vt`），并在 macOS 上提供更强的原生渲染后端（`ghostty-surface`）
- **可扩展**：与协议层（SSH/Serial/Telnet）解耦，终端只消费字节流、输出用户输入字节流
- **体验一致**：符合 `doc/组件设计规范.md` 中 Terminal 设计（黑底、光标/选区/滚动条风格）

### 1.2 约束（从现有代码得到的现实边界）

- 当前实现已经存在两条路径：
  - **macOS + `ghostty-surface`**：通过 `NSView -> CALayer.contents -> IOSurface` 抓取像素并上传到 wgpu 纹理，然后用 `egui_wgpu::Callback` 绘制（见 `src/ui/components/terminal_view.rs`）
  - **`ghostty-vt`**：通过 `ghostty_terminal_vt_write` 解析 VT，并从 render_state 拉取 codepoints grid，再用 CPU 8x8 字体绘制到 RGBA，上传纹理绘制（见 `src/backend/ghostty_vt.rs`、`src/terminal/vt_widget.rs`）
- `TerminalView` 当前同时承担：输入采集、session I/O、终端后端驱动、渲染（需要拆分）
- `src/terminal/mod.rs` 已定义 **Engine/Renderer 边界**，但尚未被 UI 层系统性采用

## 2. 终端模块的分层设计（建议落地形态）

终端模块建议拆分为 4 层，保证可测试、可替换、可演进。

```
UI(egui) TerminalView（Widget）
  ├── TerminalController（输入/焦点/剪贴板/滚动/选择/快捷键）
  ├── TerminalSessionBridge（与 AsyncSession 的字节流桥接）
  └── TerminalBackend（Engine + Renderer 组合，可切换实现）
        ├── Engine：终端状态机（写入 bytes、resize、滚动、选择状态）
        └── Renderer：把 Engine 的状态渲染到 egui（wgpu callback / egui paint）
```

### 2.1 UI Widget：`TerminalView`（建议职责收敛）

`TerminalView` 只保留“UI 层该做的事”：

- **布局与尺寸**：根据分配的 `rect` 计算 cols/rows，并触发 resize
- **输入/焦点**：将 egui 事件转为“终端输入动作”（Key/Text/Mouse/Wheel）
- **绘制入口**：调用 backend 的 `prepare/paint` 或注入 callback
- **样式应用**：背景色、滚动条、选区颜色、光标样式（遵循 `doc/组件设计规范.md`）

不建议在 `TerminalView` 内直接：

- while-loop 同步读取网络流（会卡 UI）
- 直接持有/操作 `ghostty_*` 的生命周期细节

### 2.2 TerminalController：交互与策略（统一手感）

主要负责把 UI 事件映射为终端动作，并处理 UI 策略：

- **键盘**：普通文本（IME/组合输入）、控制键、快捷键（Cmd/Ctrl/Alt 等）
- **鼠标**：点击定位/拖拽选区/双击选词（后续）/滚轮滚动
- **剪贴板**：复制选区、粘贴文本（要支持“确认读取剪贴板”策略，`ghostty_runtime_config_s` 已预留回调）
- **光标/选区风格**：
  - 光标：Idle/Focus/闪烁（视觉规范 3.2）
  - 选区：使用 Accent_Base alpha 0.25（视觉规范 3.3）
- **滚动条风格**：6px、Hover/Active 透明度（视觉规范 3.5）

### 2.3 TerminalSessionBridge：I/O 桥（与协议层解耦）

与 `crate::backend::ssh_session::AsyncSession` 对接，形成双向通道：

- **网络/PTY -> 终端**：读取到的 bytes 写入 engine（或直接写入 ghostty surface/vt）
- **用户输入 -> 网络/PTY**：将 key/text/mouse 生成的 bytes 写入 session
- **resize -> session**：当 cols/rows 变化时调用 session 的 `resize_pty(cols, rows)`（当前 `AsyncSession` trait 里已经包含此方法，见架构文档 3.3.4）

关键要求：

- **不要阻塞 UI 线程**：读 session 必须异步/后台任务化，然后通过 channel 推送到 UI
- **背压与截断策略**：避免 UI 帧内处理过多 bytes（当前代码用 `total_read > 32768 break` 做了粗略保护，需要模块化）

推荐最小实现（M0）：

- tokio task 在后台持续 read，把 bytes 发送到 `tokio::sync::mpsc::Sender<Vec<u8>>`
- UI 每帧 drain 一定数量/总字节，写入 engine，并 `request_repaint`

### 2.4 TerminalBackend：后端可切换（两条 ghostty 路线并存）

对外提供统一接口，让 UI/Controller/Bridge 不关心具体实现。

建议抽象为：

- `TerminalBackend`：组合 `Engine + Renderer`，并提供“输入编码”“渲染目标”能力
- 后端实现：
  - **Backend A：macOS 原生 `ghostty-surface`（优先性能）**
    - Engine：libghostty surface 内部状态（通过 `ghostty_surface_text`、`ghostty_surface_key` 等驱动）
    - Renderer：使用 IOSurface 像素（或更进一步：真正共享 GPU texture —— 待确认 ghostty C API 暴露度）
  - **Backend B：跨平台 `ghostty-vt`（MVP 可用）**
    - Engine：`ghostty_terminal_*` + render_state
    - Renderer：M0 采用 CPU rasterize + wgpu texture；后续替换为字体 atlas + GPU 文本渲染

## 3. 与现有代码的映射（当前在哪里、应怎么收敛）

### 3.1 现有入口：`src/ui/components/terminal_view.rs`

当前包含：

- 输入事件处理（Key/Text）
- 同步 read session 并写入终端
- 两套后端驱动（`ghostty_engine.write_ansi` / `vt_widget.on_ssh_bytes`）
- 两套渲染路径（IOSurface callback / vt callback）

建议调整目标：

- `TerminalView` → 仅负责 UI 事件采集 + 调用 controller/backend + 安排绘制
- I/O read loop → 下沉到 `TerminalSessionBridge`（tokio task）
- ghostty surface 的 IOSurface 抓取与纹理上传 → 下沉到 `Renderer`（缓存资源、避免频繁重建）

### 3.2 `src/terminal/mod.rs`：边界已存在（可作为统一 API 基础）

该文件定义了：

- `TerminalEngine`：`write/resize/scroll/query`
- `TerminalRenderer`：`prepare/paint/clear_cache`
- `TerminalSnapshot`：`size/cursor/scroll`

建议下一步把 `TerminalBackend` 基于这些 trait 组合起来，并让 `TerminalView` 只和统一接口交互。

### 3.3 `src/backend/ghostty_ffi.rs`（macOS surface）

现状：

- 负责 `ghostty_init_once`
- 封装 `GhosttyIntegration { app, surface, nsview, wakeup_cb }`
- 输入：`send_key`、`write_ansi`
- 驱动：`tick/draw`、`resize`

注意点（后续需要补齐/修正）：

- `get_cursor/get_scrollbar/get_line` 当前是占位
- IOSurface 抓取目前在 UI 文件里通过 `CALayer.contents` 获取，属于实现细节，应封装到 renderer
- `wakeup_cb` 已能触发 `egui_ctx.request_repaint()`，这对降低轮询 repaint 很关键（保留）

### 3.4 `src/backend/ghostty_vt.rs`（vt 解析）

现状：

- 完整封装 ghostty-vt 的 terminal/render_state/key_encoder
- 提供：
  - `write_vt`
  - `resize`
  - `encode_key`（用于把 key/mods 转成字节序列写入 session）
  - `snapshot_codepoints_grid_and_clear_dirty`（MVP 绘制用）

这条路径更容易跨平台落地（M0 推荐默认）。

## 4. 核心数据流（必须固定下来）

### 4.1 输出流：Session -> Terminal

```
tokio read task:
  AsyncSession::read_stream -> Vec<u8> -> mpsc -> UI thread

UI thread (每帧):
  drain bytes (限量) -> backend.engine.write(bytes)
  -> backend.renderer.prepare(snapshot/diff)
  -> backend.renderer.paint()
```

要点：

- **限量 drain**：每帧最多处理 \(N\) bytes 或 \(T\) ms，保证 UI 不掉帧
- **dirty 驱动**：`ghostty-vt` 有 dirty flag；`ghostty-surface` 有 wakeup 回调，尽量用事件驱动 repaint

### 4.2 输入流：UI -> Terminal -> Session

```
egui events -> TerminalController
  - Text: 直接 UTF-8 bytes
  - Key: backend.encode_key(...) -> bytes
  - Mouse/Scroll: (后续) -> escape seq / internal scroll

bytes -> AsyncSession::write_stream(bytes)
```

要点：

- `Text` 与 `Key` 要区分：`Text` 由 IME/输入法产生，优先走 text；`Key` 走 encoder
- `ghostty-vt` 已提供 `encode_key`，应成为“统一输入编码”的基石
- `ghostty-surface` 的 `send_key` 可能不需要转 bytes（直接喂 surface），但仍需将“实际发送给远端的 bytes”保持一致：**最终远端看到的输入必须一致**（否则本地渲染和远端状态会分叉）

推荐策略：

- **以 Session 为真相**：所有输入都写入 session；终端显示由 session 回显驱动
- 本地对 `ghostty-surface` 的直接写入仅用于“本地回显优化”时才启用（M1 以后再做）

## 5. 后端选择策略（编译期 feature + 运行时 fallback）

结合 `src/backend/mod.rs` 和当前 feature：

- `ghostty-surface`：macOS + `feature = "ghostty-surface"`
- `ghostty-vt`：`feature = "ghostty-vt"`（跨平台）

建议默认策略：

- **优先 `ghostty-vt` 作为通用后端**（M0 可靠）
- **macOS 且启用 `ghostty-surface` 时可切到原生后端**（M1）
- 运行时如果 surface 初始化失败（`nsview`/layer/contents 获取不到），自动 fallback 到 `ghostty-vt`

## 6. MVP（M0）开发里程碑与验收标准

### 6.1 M0：可用终端（跨平台）

范围：

- 只启用 `ghostty-vt` 后端（或作为默认）
- 后台异步 read session，UI 每帧 drain
- 文字显示正确（基本 ASCII + 常见 ANSI 颜色可后置）
- 输入：Enter/Backspace/Arrow/Tab/Esc + 普通 Text
- resize：窗口变化触发 cols/rows 更新，并调用 `resize_pty`

验收：

- 连接真实 SSH（或 mock_session）执行 `ls`, `top`, `vim`（vim 可能需要 M1 才顺畅）
- UI 不因输出爆发卡死（例如 `yes | head -n 10000`）

### 6.2 M1：体验增强（与设计规范对齐）

范围：

- 光标形态/闪烁（遵循规范 3.2）
- 自定义选区颜色（规范 3.3），并禁止系统默认蓝色选区
- 滚动条样式（规范 3.5），支持滚轮滚动与 scrollback
- 更合理的字体与行高（规范 3.1：JetBrains Mono / SF Mono、字号 13/14、行高 1.4）

验收：

- 选区视觉一致、滚动顺滑、长时间输出不掉帧明显

### 6.3 M2：macOS 原生渲染后端（性能/质量）

范围：

- `ghostty-surface` 后端工程化：
  - renderer 封装 IOSurface 抓取与纹理上传
  - 资源缓存与尺寸变化重建
  - 使用 wakeup_cb 驱动 repaint
- 输入、resize、剪贴板回调逐步补齐

验收：

- macOS 下高吞吐输出（例如 `cat` 大文件）仍稳定 60fps 附近

## 7. 关键接口草案（用于后续开发对齐）

> 目标是让 `TerminalView` 只依赖统一接口，而不是分支 ifdef 大块逻辑。

### 7.1 统一后端接口（建议新增：`src/terminal/backend.rs`）

建议接口形态（草案）：

- `TerminalBackend::resize(cols, rows)`
- `TerminalBackend::ingest_output(bytes)`（来自 session）
- `TerminalBackend::handle_event(event, modifiers) -> Vec<Vec<u8>>`（产出要写入 session 的 bytes）
- `TerminalBackend::paint(ui, rect)`（或返回 egui callback）

实现上：

- `ghostty-vt` 后端：`handle_event` 调 `encode_key`，`paint` 使用现有 `VtTerminalWidget` 渲染策略（M0 先复用，M1/M2 再替换）
- `ghostty-surface` 后端：`handle_event` 先产出 session bytes（真相），是否额外喂给 surface 作为本地预览后续决定

### 7.2 I/O 桥（建议新增：`src/terminal/session_bridge.rs`）

最小字段：

- `rx_from_session: mpsc::Receiver<Vec<u8>>`
- `tx_to_session: mpsc::Sender<Vec<u8>>`
- `spawn_read_loop(session)`：后台 read_stream
- `spawn_write_loop(session)`：后台写入（或 UI 线程直接写入也可，但建议统一）

## 8. 性能与正确性清单（开发过程中必须持续检查）

### 8.1 UI 不阻塞

- 禁止在 `TerminalView::render` 内做“无限 while read_stream”
- 每帧处理输出要限量（bytes 上限、或时间片上限）

### 8.2 纹理上传成本可控

- wgpu texture / pipeline / bind group 必须缓存，尺寸变化才重建
- 像素数据上传要对齐 `bytes_per_row`（现有 IOSurface 路径已处理 256 对齐）

### 8.3 输入一致性

- 以 session 回显驱动最终显示，避免“本地写入 ghostty，但远端没收到”造成分叉

## 9. 测试建议（从低到高）

- **单元测试**：
  - 输入编码：`ghostty-vt encode_key` 对常用按键生成序列非空且符合预期
  - resize：cols/rows 变化时仅触发一次 resize 调用
- **集成测试**：
  - `mock_session`：模拟输出爆发、延迟、分片（4096/不规则）
- **手工测试**：
  - `ssh`：运行 `top`/`htop`、滚动、复制粘贴、窗口 resize

## 10. 本文档与当前代码的“下一步落地动作”（建议顺序）

> 这部分是给开发者的具体落地路线图（不依赖额外架构讨论）。

- 先以 `ghostty-vt` 做 **M0 可用**：把 session read loop 下沉到 bridge，UI 每帧 drain
- 把 `TerminalView` 拆成：`TerminalController + TerminalBackend + SessionBridge`
- 把 macOS IOSurface 相关渲染逻辑迁移到 renderer，避免 UI 文件膨胀
- 在 M1 再做视觉一致性（光标/选区/滚动条/字体）
- 在 M2 再把 `ghostty-surface` 的能力补齐（剪贴板回调、selection、scrollbar/cursor 查询）
