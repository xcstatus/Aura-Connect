## 2. 根本原因

发现了 **两个独立的 Bug**，都导致数据丢失：

### Bug 1: `SshChannel::read_stream` 缓冲区清空

**位置**: `src/backend/ssh_session.rs`

**问题代码**:
```rust
fn read_stream(&mut self, buffer: &mut [u8]) -> Result<usize> {
    if !self.read_buffer.is_empty() {
        let len = std::cmp::min(buffer.len(), self.read_buffer.len());
        buffer[..len].copy_from_slice(&self.read_buffer[..len]);
        self.read_buffer = Vec::new();  // BUG: 直接清空整个缓冲区！
        return Ok(len);
    }
    // ...
}
```

**Bug 场景**:
1. `read_buffer` 包含 20KB 数据（多行文本）
2. `buffer` 只有 4KB
3. 读取了前 4KB
4. `Vec::new()` 清空了全部 20KB 缓存
5. 剩余 16KB 数据永久丢失

---

### Bug 2: `BaseSshConnection::read_channel` 溢出丢弃

**位置**: `src/backend/ssh_session.rs`

**问题代码**:
```rust
async fn read_channel(&self, channel_id: ChannelId, buffer: &mut [u8]) -> Result<usize> {
    match rx.try_recv() {
        Ok(data) => {
            let len = std::cmp::min(buffer.len(), data.len());
            buffer[..len].copy_from_slice(&data[..len]);
            if data.len() > len {
                // TODO: 缓存超出的数据  // BUG: 超出部分直接丢弃！
            }
            Ok(len)
        }
        // ...
    }
}
```

**Bug 场景**:
1. mpsc 接收到 20KB 数据包
2. `buffer` 只有 4KB
3. 读取了前 4KB
4. 剩余 16KB 数据直接丢弃

## 3. 修复方案

### Bug 1 修复: 保留未读取的缓冲区字节

```rust
fn read_stream(&mut self, buffer: &mut [u8]) -> Result<usize> {
    if !self.read_buffer.is_empty() {
        let len = std::cmp::min(buffer.len(), self.read_buffer.len());
        buffer[..len].copy_from_slice(&self.read_buffer[..len]);
        // 修复：保留未读取的尾部字节
        if len < self.read_buffer.len() {
            let remaining = self.read_buffer[len..].to_vec();
            self.read_buffer = remaining;
        } else {
            self.read_buffer.clear();
        }
        return Ok(len);
    }
    // ...
}
```

### Bug 2 修复: 添加溢出缓冲区

**Step 1**: `ChannelEntry` 添加字段
```rust
struct ChannelEntry {
    data_rx: Arc<tokio::sync::Mutex<mpsc::Receiver<Vec<u8>>>>,
    cmd_tx: mpsc::UnboundedSender<SessionCmd>,
    overflow_buf: Arc<tokio::sync::Mutex<Vec<u8>>>,  // 新增
}
```

**Step 2**: `open_channel` 创建溢出缓冲区
```rust
let overflow_buf = Arc::new(tokio::sync::Mutex::new(Vec::new()));
let entry = ChannelEntry {
    data_rx,
    cmd_tx,
    overflow_buf: overflow_buf.clone(),
};
```

**Step 3**: `read_channel` 实现溢出逻辑
```rust
async fn read_channel(&self, channel_id: ChannelId, buffer: &mut [u8]) -> Result<usize> {
    // 优先从溢出缓冲区读取
    {
        let mut overflow = entry.overflow_buf.lock().await;
        if !overflow.is_empty() {
            let len = std::cmp::min(buffer.len(), overflow.len());
            buffer[..len].copy_from_slice(&overflow[..len]);
            if len < overflow.len() {
                overflow.drain(..len);
            } else {
                overflow.clear();
            }
            return Ok(len);
        }
    }

    // 从 mpsc 接收数据，超出部分存入溢出缓冲区
    match rx.try_recv() {
        Ok(data) => {
            let len = std::cmp::min(buffer.len(), data.len());
            buffer[..len].copy_from_slice(&data[..len]);
            if data.len() > len {
                let remaining = data[len..].to_vec();
                let mut overflow = entry.overflow_buf.lock().await;
                overflow.extend_from_slice(&remaining);
            }
            Ok(len)
        }
        // ...
    }
}
```

## 6. 经验总结

### 6.1 数据完整性原则

1. **边界处理**: 当读取的数据量小于可用数据量时，必须保留未读取的部分
2. **缓冲区语义**: `read` 操作不应改变未读取数据的状态
3. **溢出缓存**: 当目标缓冲区小于源数据时，应有明确的溢出处理策略

### 6.2 诊断方法

1. **日志分析**: 观察 `total_rows` 的增长是否符合预期
2. **行号连续性**: 检查输出行号是否有跳跃
3. **内容拼接错误**: 观察是否有"行末尾+行开头"的拼接现象

### 6.3 代码审查要点

- 检查所有 `buffer` / `read` / `recv` 相关代码的边界处理
- 确认"未读取部分"有明确的保留或缓存策略
- 特别注意 `Vec::new()` / `clear()` / `drain()` 等操作是否会丢弃未处理数据
